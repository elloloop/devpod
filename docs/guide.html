devpod Documentation

The complete guide to devpod — stacked diffs, local CI, version history, and a diff viewer powered by VS Code. Everything you need to ship code faster.

Getting Started #

devpod is a developer workflow CLI. It wraps git with a simple set of commands built around stacked diffs, local CI, and version history. You can be productive in under 5 minutes.

Install #

Install devpod with a single command. No package manager required.

Terminal 
$ curl -fsSL https://raw.githubusercontent.com/elloloop/devpod/main/install.sh | sh

Supported platforms: macOS (Apple Silicon and Intel) and Linux (x86_64 and ARM64). The binary is 6.6MB with zero dependencies.

Verify the installation:

Terminal 
$ devpod --version
devpod version 0.x.x

Clone a Repo #

Clone any GitHub repository. devpod sets up the repo with its own metadata so it can track features, diffs, and versions.

Terminal 
$ devpod clone myorg/myrepo
Cloning into 'myrepo'...
✓ Repository ready
Tip

You can also use devpod in an existing git repository. Just navigate to the repo and run any devpod command — it will initialize automatically.

Create Your First Feature #

A feature is a unit of work. It maps to a git branch underneath, but you don't need to think about branches.

Terminal 
$ devpod feature "add user authentication"
✓ Started feat: add user authentication

Now make some code changes. Edit files as you normally would.

Save Your First Diff #

A diff is a snapshot of your changes. Think of it as a commit + PR combined. devpod stages all your changes and creates a diff.

Terminal 
$ devpod diff "add auth API endpoints"
✓ Created D1: feat(auth): add auth API endpoints
  +200/-0 · 3 files

If you omit the message, devpod uses an LLM to generate a title from your changes (if configured). You can also just run devpod diff with no arguments.

Submit for Review #

Submit pushes your diffs to GitHub as pull requests and opens the local dashboard for review.

Terminal 
$ devpod submit
✓ Submitted 1 diff
  Review: http://localhost:3000/diffs/add-user-authentication

Land (Ship It) #

Land merges your approved diffs to the main branch.

Terminal 
$ devpod land
✓ Landed D1: feat(auth): add auth API endpoints
  Feature complete!

That's the entire basic workflow: feature → diff → submit → land.

Core Concepts #

Features #

A feature is a unit of work — a bug fix, a new feature, a refactor, a documentation update. Under the hood, it maps to a git branch, but devpod manages the branch for you.

Create features with different prefixes depending on the type of work:

Terminal 
$ devpod feature "add search"        # feat/add-search
$ devpod fix "login timeout"        # fix/login-timeout
$ devpod docs "update API reference" # docs/update-api-reference
$ devpod chore "upgrade dependencies"# chore/upgrade-dependencies

You can have multiple features in progress. Use devpod features to list them and devpod switch to move between them.

Diffs #

A diff is a reviewable unit of change within a feature. One feature can have multiple diffs, forming a stack. Each diff becomes its own GitHub PR.

  • D1, D2, D3... — diffs are numbered sequentially within a feature
  • Stacking — each subsequent diff builds on the previous one
  • Versioning — every time you update a diff, a new version (v1, v2, v3...) is saved
  • Auto-rebase — when you edit a diff in the middle of a stack, all subsequent diffs rebase automatically

Sync #

Sync pulls the latest changes from the main branch and rebases your diffs on top. Run it regularly to stay up to date and avoid conflicts.

Terminal 
$ devpod sync
✓ Synced with latest code
Important

Always run devpod diff to save your changes before devpod sync. Sync rebases your work and needs a clean working directory.

Submit #

Submit pushes your diffs to GitHub as pull requests and opens the devpod dashboard for review. Each diff in the stack becomes a separate PR, linked together.

Land #

Land merges your diffs to the main branch. Diffs are landed in order (D1 first, then D2, etc.). Each diff is squash-merged into a clean commit.

The Versions Branch #

devpod stores version history in a special devpod/versions branch. Every time you create or update a diff, the full state is saved as a version. This enables:

  • True interdiff — reviewers can see what changed between v1 and v2 of a diff, not just the full diff
  • History browsing — you can go back to any version of any diff
  • Audit trail — every change is recorded, nothing is lost

Commands Reference #

Complete reference for every devpod command. All commands should be run from within a devpod-managed repository.

devpod clone #

Clone a GitHub repository and initialize devpod tracking.

Terminal 
$ devpod clone myorg/myrepo
$ devpod clone https://github.com/myorg/myrepo.git

devpod feature / fix / docs / chore / start #

Start a new unit of work. Each variant sets a conventional commit prefix.

Terminal 
$ devpod feature "add user search"     # prefix: feat
$ devpod fix "null pointer in auth"     # prefix: fix
$ devpod docs "add API examples"        # prefix: docs
$ devpod chore "update CI config"       # prefix: chore
$ devpod start "custom-prefix" "desc"  # custom prefix

devpod diff #

Create or update a diff. This is the primary command you'll use — it stages all changes and creates a versioned snapshot.

Usage 
# Create a new diff with a message
$ devpod diff "add login API"

# Create a diff with auto-generated title (requires LLM config)
$ devpod diff

# Edit an existing diff (switch to it for editing)
$ devpod diff edit D1

# After editing, save changes (creates a new version)
$ devpod diff

# Continue after resolving conflicts
$ devpod diff --continue

# Abort an in-progress operation
$ devpod diff --abort

# Preview what would be included in the diff
$ devpod diff --preview

# View version history of a diff
$ devpod diff versions D1

# Compare two versions of a diff
$ devpod diff compare D1:v1 D1:v2

# Check diff status (CI results, review status)
$ devpod diff check

devpod sync #

Pull latest changes from the main branch and rebase your diffs on top.

Terminal 
$ devpod sync

# Continue after resolving conflicts
$ devpod sync --continue

# Abort a sync in progress
$ devpod sync --abort

devpod switch #

Switch between features. Shows an interactive picker if no argument is given.

Terminal 
$ devpod switch                    # interactive picker
$ devpod switch feat/add-search     # switch directly

devpod submit #

Push diffs to GitHub as PRs and open the dashboard.

Terminal 
$ devpod submit

# Preview what would be submitted without pushing
$ devpod submit --preview

devpod land #

Merge approved diffs to the main branch.

Terminal 
$ devpod land

# Force land without approval checks
$ devpod land --force

devpod features #

List all features in progress.

Terminal 
$ devpod features
  * feat/add-search      (2 diffs, current)
    fix/login-timeout    (1 diff)
    docs/api-reference   (0 diffs)

devpod diffs #

List all diffs in the current feature.

Terminal 
$ devpod diffs
  D1  feat(auth): add API endpoints    v2  +200/-10  3 files
  D2  feat(auth): add login form       v1  +150/-0   2 files

devpod context #

Show current context: which feature you're on, which diff, and the state of things.

Terminal 
$ devpod context
  Feature:  feat/add-search
  Diff:     D2 (latest)
  Branch:   feat/add-search
  Base:     main
  Diffs:    2 (D1 submitted, D2 in progress)

devpod status #

Show uncommitted changes in the working directory.

Terminal 
$ devpod status
  Modified:  src/auth/api.go
  Added:     src/auth/middleware.go
  2 files changed

devpod log #

Show the history of diffs and versions for the current feature.

Terminal 
$ devpod log

devpod undo #

Undo the last operation. Keeps a history you can browse.

Terminal 
$ devpod undo            # undo last operation
$ devpod undo --list     # show undo history

devpod split #

Split a diff into multiple smaller diffs. Useful when a diff grows too large.

Terminal 
$ devpod split

devpod config #

Get and set configuration values.

Terminal 
$ devpod config set llm.provider anthropic
$ devpod config set llm.api_key sk-ant-...
$ devpod config get llm.provider
$ devpod config set ci.auto_run true
$ devpod config set default_branch main

devpod runner #

Manage the local CI runner daemon.

Terminal 
$ devpod runner start    # start the daemon
$ devpod runner stop     # stop the daemon
$ devpod runner status   # check if running

devpod run #

Run a specific workflow locally.

Terminal 
$ devpod run ci.yml          # run a specific workflow
$ devpod run ci.yml --job test # run a specific job

devpod runs #

List recent CI runs.

Terminal 
$ devpod runs

devpod workflows #

List available GitHub Actions workflows in the repository.

Terminal 
$ devpod workflows
  ci.yml          CI Pipeline
  deploy.yml      Deploy to Production
  lint.yml        Lint and Format

devpod secret #

Manage secrets for local CI runs. Secrets are stored encrypted on your machine.

Terminal 
$ devpod secret set NPM_TOKEN         # prompts for value
$ devpod secret list                   # list secret names
$ devpod secret delete NPM_TOKEN       # delete a secret
$ devpod secret import .env            # import from .env file
$ devpod secret sync                   # sync from GitHub secrets
$ devpod secret status                 # show which secrets are set

devpod dashboard #

Open the web dashboard for reviewing diffs, viewing CI results, and browsing version history.

Terminal 
$ devpod dashboard
  Dashboard: http://localhost:3000

Workflows #

Common workflows for different scenarios. These show how devpod commands compose together in real-world usage.

Solo Developer Workflow #

When you're working alone on a project:

Solo Workflow 
$ devpod clone myorg/myrepo
$ cd myrepo
$ devpod feature "add search"

# make changes...
$ devpod diff "add search backend"

# make more changes...
$ devpod diff "add search UI"

# ready to ship
$ devpod submit
$ devpod land

Team Workflow #

When working with a team. Your teammates don't need devpod — they review on GitHub as usual.

Team Workflow 
$ devpod feature "add payment processing"

# first piece: the API
$ devpod diff "add payment API"

# second piece: the webhook handler
$ devpod diff "add webhook handler"

# submit both for review (creates 2 linked PRs)
$ devpod submit

# teammate reviews on GitHub, requests changes to D1
$ devpod diff edit D1
# make changes...
$ devpod diff

# re-submit (D2 auto-rebases)
$ devpod submit

# after approval
$ devpod land

Open Source Contributor Workflow #

Contributing to an open source project you don't own:

Open Source Workflow 
$ devpod clone someorg/cool-project
$ cd cool-project
$ devpod fix "handle edge case in parser"

# make the fix
$ devpod diff "fix off-by-one in token parser"

# submit creates a PR from your fork
$ devpod submit

Stacking Multiple Diffs #

The real power of devpod. Break a large feature into small, reviewable pieces:

Stacking Example 
$ devpod feature "user profiles"

# D1: database schema
$ devpod diff "add user profile schema"

# D2: API endpoints (builds on D1)
$ devpod diff "add profile API endpoints"

# D3: frontend (builds on D2)
$ devpod diff "add profile page UI"

# Submit all 3 as linked PRs
$ devpod submit

# Reviewer can approve D1 independently
# Land D1 first, D2 and D3 auto-rebase
$ devpod land

Editing a Diff in the Middle of a Stack #

One of devpod's killer features. Edit any diff in the stack, and subsequent diffs rebase automatically:

Editing a Stack 
# You have D1, D2, D3. Reviewer wants changes to D1.
$ devpod diff edit D1

# make changes to D1...

$ devpod diff
✓ Updated D1 → v2
  D2 rebased automatically.
  D3 rebased automatically.

Resolving Conflicts #

Conflicts can happen during devpod sync or when editing a diff in a stack. devpod guides you through resolution:

Conflict Resolution 
$ devpod sync
  Conflict in src/auth/api.go

# Edit the file, remove conflict markers
# (<<<<<<<, =======, >>>>>>>)

$ devpod sync --continue
✓ Synced successfully

# Or abort if you want to start over:
$ devpod sync --abort

Working Offline #

devpod works fully offline. You can create features, save diffs, edit stacks, and run local CI without a network connection. Just devpod submit when you're back online.

Offline Workflow 
# All of these work offline:
$ devpod feature "offline work"
$ devpod diff "add feature"
$ devpod diff edit D1
$ devpod diff
$ devpod diffs
$ devpod context
$ devpod run ci.yml

# When back online:
$ devpod sync
$ devpod submit

The Dashboard #

devpod includes a web-based dashboard for reviewing diffs. It uses the Monaco Editor (the same engine as VS Code) for syntax-highlighted, side-by-side diff viewing.

Reviewing Diffs #

Open the dashboard after submitting:

Terminal 
$ devpod dashboard
  Dashboard: http://localhost:3000

The dashboard shows:

  • All diffs in the current feature, with their stack order
  • File changes for each diff with syntax highlighting
  • Side-by-side or unified diff view
  • CI run results
  • Version history for each diff

Version History #

Every time you update a diff, a new version is saved. The dashboard lets you compare any two versions — true interdiff. This is especially valuable for reviewers who want to see "what changed since my last review?" instead of re-reading the entire diff.

Keyboard Shortcuts #

Shortcut Action
j / kNavigate between files
n / pNext / previous diff in stack
sToggle side-by-side / unified
vOpen version picker
tToggle theme (dark / light / system)
?Show keyboard shortcuts

Local CI Runner #

devpod can run your GitHub Actions workflows locally. Same YAML files, same marketplace actions. Catch bugs in seconds on your machine instead of waiting for GitHub's CI.

Starting the Runner #

The runner is a background daemon that watches for changes and runs workflows automatically (if configured).

Terminal 
$ devpod runner start
✓ Runner started (PID 12345)

$ devpod runner status
  Runner: running (PID 12345)

$ devpod runner stop
✓ Runner stopped

Running Workflows #

Run workflows manually or let the runner trigger them automatically on each diff.

Terminal 
# List available workflows
$ devpod workflows

# Run a specific workflow
$ devpod run ci.yml

# Run a specific job within a workflow
$ devpod run ci.yml --job test

# View recent runs
$ devpod runs

# Enable auto-run (runs CI on every diff)
$ devpod config set ci.auto_run true

Secrets Management #

Your workflows likely need secrets (API keys, tokens). devpod stores these encrypted on your machine.

Terminal 
# Set a secret (prompts for value)
$ devpod secret set NPM_TOKEN

# Import secrets from a .env file
$ devpod secret import .env

# Sync from GitHub repository secrets
$ devpod secret sync

# List configured secrets
$ devpod secret list
  NPM_TOKEN       (set)
  GITHUB_TOKEN    (set)
  AWS_ACCESS_KEY  (not set)

# Check status of all required secrets
$ devpod secret status
How it works

devpod reads your .github/workflows/*.yml files, parses the workflow definitions, and executes them locally using Docker containers (when available) or direct process execution. Marketplace actions are fetched and cached locally.

Configuration #

devpod configuration is stored per-repository. Use devpod config to manage it.

LLM Setup #

devpod can use an LLM to auto-generate diff titles from your code changes. Supported providers:

Provider Configuration
Anthropic devpod config set llm.provider anthropic
devpod config set llm.api_key sk-ant-...
OpenAI devpod config set llm.provider openai
devpod config set llm.api_key sk-...
Ollama (local) devpod config set llm.provider ollama
devpod config set llm.model codellama
Custom devpod config set llm.provider custom
devpod config set llm.endpoint https://...

CI Auto-Run #

Enable automatic CI runs whenever you create or update a diff:

Terminal 
$ devpod config set ci.auto_run true

Default Branch #

Set the default branch if your repo uses something other than main:

Terminal 
$ devpod config set default_branch develop

For AI Agents #

Agent Instructions

This section is designed to be read by AI coding agents (Claude, GPT, Copilot, etc.) to enable them to use devpod on behalf of a developer.

Quick Start for Agents

Agent Protocol 
# 1. Check if devpod is installed
$ devpod --version

# 2. If not installed:
$ curl -fsSL https://raw.githubusercontent.com/elloloop/devpod/main/install.sh | sh

# 3. Navigate to the repository
$ cd /path/to/repo

# 4. Start a feature
$ devpod feature "description of work"

# 5. Make code changes

# 6. Create a diff
$ devpod diff "what you did"

# 7. If more changes needed, repeat steps 5-6
# (this creates stacked diffs: D1, D2, D3...)

# 8. Submit for review
$ devpod submit

# 9. After approval
$ devpod land

Key Rules for Agents

  • Always devpod diff before devpod sync — save changes first, then sync. Sync requires a clean working directory.
  • One logical change per diff — keep diffs small and focused. If you're making multiple unrelated changes, use separate diffs.
  • Use conventional commit style for diff messages — e.g., "add user validation", "fix timeout in auth handler", "update API docs".
  • Check devpod context if unsure where you are — it shows the current feature, diff, and state.
  • Check devpod status to see uncommitted changes — this tells you if there are changes that need to be saved as a diff.
  • Use devpod diff --preview to see what would be included before creating a diff.
  • If an operation fails, check if you need --continue or --abort to resolve an interrupted operation.

Agent Decision Tree

Decision Logic 
# Starting work?
devpod feature "description" → make changes → devpod diff "message"

# More changes to add?
make changes → devpod diff "message"  # creates D2, D3, etc.

# Need to edit a previous diff?
devpod diff edit D1 → make changes → devpod diff

# Need latest code from main?
devpod diff → devpod sync  # always save first!

# Ready for review?
devpod submit

# Ready to merge?
devpod land

# Conflict during sync?
edit files → devpod sync --continue

# Interrupted operation?
devpod diff --continue  OR  devpod diff --abort

# Not sure what state you are in?
devpod context
devpod status

Troubleshooting #

Common issues and how to fix them.

"You have unsaved changes"

You have modified files that aren't saved as a diff yet. Run devpod diff to save them before proceeding.

Fix 
$ devpod diff

"A previous operation was interrupted"

A rebase, sync, or diff edit was interrupted. You need to either continue or abort.

Fix 
$ devpod diff --continue    # finish the operation
# or
$ devpod diff --abort       # discard and start over

"Not on a feature branch"

You're on the main branch or a branch devpod doesn't recognize. Start a new feature first.

Fix 
$ devpod feature "my work"

Conflicts during sync

Your changes conflict with updates from the main branch. Edit the conflicting files, remove the conflict markers (<<<<<<<, =======, >>>>>>>), then continue.

Fix 
# 1. Edit the conflicting files
# 2. Remove conflict markers
# 3. Continue:
$ devpod sync --continue

Dashboard not loading

Make sure the runner is started and the dashboard command has been run.

Fix 
$ devpod runner start
$ devpod dashboard

Need to undo something

devpod tracks operations so you can undo them.

Fix 
$ devpod undo --list    # see what can be undone
$ devpod undo            # undo last operation
